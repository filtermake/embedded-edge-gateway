**文档版本**:1.0
**适用模块**:Project 1 (网关) M5 协议解析器 / Project 2 (STM32 节点) 帧发送
**最后更新**:2026-05-27

---

## 1. 概述

### 1.1 目的

本协议定义 STM32 传感器节点(Project 2)与树莓派网关(Project 1)之间通过 UART 串口传输应用层数据的帧格式与解析规则。

### 1.2 设计目标

按优先级排序:

1. **健壮性**:嵌入式现场存在电磁干扰,协议需检测并丢弃损坏帧,不能让接收端 FSM 卡死
2. **简洁性**:STM32 资源紧张(64KB Flash / 20KB RAM),协议实现不能过重
3. **可观测性**:解析错误需可记录、可定位,运维友好
4. **效率**:慢速场景(≤ 10 Hz),字节开销不是瓶颈,可适当冗余换健壮性

### 1.3 设计哲学

本协议属于"**标记流派**" —— 用固定帧头标识帧边界,而非依赖时序(对比:Modbus RTU 属于"时序流派",靠 3.5 字符静默分帧)。

理由:UART 异步通信无严格时序约束,标记流派更通用;且接收端可以是 Linux 用户态程序,无法保证字符级时序精度。

### 1.4 基本约定

- **字节序**:多字节字段(本协议仅 CRC16)使用**小端字节序 (Little Endian)**,低字节在前
- **数值表示**:文档中所有十六进制值用 `0x` 前缀,如 `0xAA`
- **单帧最大长度**:69 字节(`2 + 1 + 64 + 2`)
- **单帧最小长度**:6 字节(心跳包:`2 + 1 + 1 + 2`)

## 2. 帧格式

### 2.1 完整帧结构

```
+--------+--------+------+------+------------------+--------+--------+
| 帧头1  | 帧头2  | LEN  | TYPE | payload data     | CRC_LO | CRC_HI |
+--------+--------+------+------+------------------+--------+--------+
  1 字节   1 字节   1B     1B     (LEN-1) 字节         1B       1B
```

### 2.2 字段表

| 字段 | 大小 | 取值范围 | 是否参与 CRC | 含义 |
|---|---|---|---|---|
| 帧头1 | 1 字节 | `0xAA`(固定) | ❌ 否 | 帧起始标识(高位 10101010) |
| 帧头2 | 1 字节 | `0x55`(固定) | ❌ 否 | 帧起始标识(高位 01010101) |
| LEN | 1 字节 | 1 ~ 64 | ✅ 是 | TYPE 字节 + payload data 字节的总长度 |
| TYPE | 1 字节 | 0x01 ~ 0xFE | ✅ 是 | 业务数据类型标识 |
| payload data | (LEN - 1) 字节 | 任意 | ✅ 是 | 业务数据,格式由 TYPE 决定 |
| CRC_LO | 1 字节 | 0x00 ~ 0xFF | ❌ 否 | CRC16 低字节 |
| CRC_HI | 1 字节 | 0x00 ~ 0xFF | ❌ 否 | CRC16 高字节 |

### 2.3 CRC 覆盖范围

```
+--------+--------+------+------+------------------+--------+--------+
|  不含  |  不含  | LEN  | TYPE | payload data     |  不含  |  不含  |
+--------+--------+------+------+------------------+--------+--------+
                  └─────── CRC 计算范围 ──────────┘
```

**理由**:
- 帧头是固定值,损坏由 FSM 在头部识别阶段直接丢弃,不进入 CRC 计算
- CRC 字段本身不参与自校验
- 减少 CRC 计算量(嵌入式哲学:能省则省)

## 3. 字段详解

### 3.1 帧头(2 字节)

**固定值**:`0xAA 0x55`

**为什么是这个值**:二进制 `10101010 01010101`,位变化频繁,接收端 UART 时钟同步友好。这是工业标配(大疆 SDK、激光雷达、串口屏均使用)。

**为什么用 2 字节而不是 1 字节**:
- 1 字节帧头假阳性率 = 1/256 ≈ 0.39%(payload 中恰好出现帧头字节)
- 2 字节帧头假阳性率 = 1/65536 ≈ 0.0015%(降低 256 倍)
- 4 字节帧头更安全但过度设计,M5 场景不需要

### 3.2 LEN(1 字节)

**含义**:LEN = TYPE 字段长度(1B)+ payload data 长度

**合法取值**:1 ≤ LEN ≤ 64

**LEN 取值示例**:
- LEN = 1:心跳包(只有 TYPE,无 payload data)
- LEN = 2:状态包(TYPE + 1 字节状态)
- LEN = 3:光照包(TYPE + 2 字节光照值)
- LEN = 6:DHT11 温湿度包(TYPE + 5 字节温湿数据)

**LEN_MAX = 64 的推导**:
1. 列出业务上所有 TYPE 的 payload 长度,最大 33 字节(复合环境包 + 扩展余量)
2. 给 ~2 倍余量,向上取整到 2 的幂 → 64
3. 不选 128 是为了降低 LEN 损坏后落在合法范围的概率(防灾难传播)
4. 不选 32 是为了留扩展空间

**LEN_MAX 的可变性**:此值与业务挂钩,业务扩展(如新增 100 字节 payload 的 TYPE)时需重新评估。代码中以 `constexpr uint8_t LEN_MAX = 64;` 命名常量定义。

### 3.3 TYPE(1 字节)

**含义**:业务数据类型标识

**TYPE 字典(v1.0)**:

| TYPE | 含义 | payload data 长度 | LEN |
|---|---|---|---|
| `0x00` | **保留(非法)** | — | — |
| `0x01` | DHT11 温湿度 | 5 字节(温度 2B + 湿度 2B + 校验 1B) | 6 |
| `0x02` | BH1750 光照 | 2 字节(光照值,大端 uint16) | 3 |
| `0x03` | 心跳包 | 0 字节 | 1 |
| `0x04` | 设备状态 | 1 字节(状态枚举) | 2 |
| `0x05` ~ `0x0F` | 预留 | — | — |
| `0x10` | 复合环境包(v1.1+) | ≤ 16 字节 | ≤ 17 |
| `0x11` ~ `0xEF` | 预留 | — | — |
| `0xF0` ~ `0xFE` | 厂商扩展预留 | — | — |
| `0xFF` | **保留(非法)** | — | — |

**为什么 0x00 和 0xFF 保留**:这两个值是"全 0"和"全 1",最容易因电气问题(空线/短路)被误读出来,作为非法 TYPE 用于检测异常。

**TYPE 字段的校验责任**:协议层(M5 FrameParser)**不验证 TYPE 是否在字典中**,只保证帧的物理结构合法。TYPE 字典的合法性由上层应用检查。这是关注点分离(Separation of Concerns)的设计 —— 协议层稳定,业务层迭代不影响协议层。

### 3.4 payload data(变长)

**格式**:由 TYPE 决定,本协议层不规定。

**长度**:`LEN - 1` 字节。LEN = 1 时无 payload data。

**示例**(DHT11 温湿度,TYPE = 0x01):

```
payload data = [温度高字节] [温度低字节] [湿度高字节] [湿度低字节] [校验和]
                ↑─────── 温度 (×10) ───────↑ ↑──── 湿度 (×10) ────↑
                例:25.3°C → 0x00FD (= 253)   例:60.5% → 0x025D (= 605)
```

**字节序约定**:payload data 内部使用**大端字节序**(网络字节序,符合传感器原始数据习惯)。

## 4. 校验算法

### 4.1 算法参数

| 参数 | 值 |
|---|---|
| 算法 | CRC16-MODBUS |
| 多项式 | `0xA001`(= `0x8005` 反射) |
| 初始值 | `0xFFFF` |
| 反射输入 | 是 |
| 反射输出 | 是 |
| 末位异或 | 无(XOR 0x0000) |

**为什么选 CRC16-MODBUS**:
- **嵌入式串口事实标准**(Modbus RTU 用相同算法)
- 对 ≤ 1KB 的帧检测能力优秀(可检测所有 ≤ 16 位的突发错误)
- 计算开销小(查表法每字节 ~10 个时钟周期)
- 不选 CRC32 是因为 M5 帧最大 69 字节,CRC32 过度设计且多占 2 字节

### 4.2 算法实现

推荐使用**查找表法**(空间换时间,256 项 × 2 字节 = 512 字节查找表):

```c
static const uint16_t kCrc16Table[256] = { /* 256 项预算值 */ };

uint16_t crc16_update(uint16_t crc, uint8_t byte) {
    return (crc >> 8) ^ kCrc16Table[(crc ^ byte) & 0xFF];
}
```

### 4.3 测试向量(必须通过)

发送/接收两端必须使用以下测试向量验证 CRC 实现一致性。**只要任一向量不匹配,实现错误,绝不能上线**。

| 输入字节(hex) | 期望 CRC16(uint16_t) | CRC_LO | CRC_HI |
|---|---|---|---|
| `01 03 00 00 00 02` | `0x0BC4` | `0xC4` | `0x0B` |
| `01 06 00 01 00 03` | `0x0B98` | `0x98` | `0x0B` |
| `02 01 00 00 00 02` | `0xF8BD` | `0xBD` | `0xF8` |
| `04` | `0x83BE` | `0xBE` | `0x83` |

**外部验证渠道**:可用 https://crccalc.com/ 选 "CRC-16/MODBUS" 算法,输入相同字节比对(注意网站可能按"线上字节顺序"显示,需做字节交换比对)。

### 4.4 在帧中的字节顺序

CRC16 的 16 位值按**小端**写入帧尾:

```
计算得 CRC = 0x1234
帧尾两字节 = [0x34] [0x12]
              CRC_LO  CRC_HI
```

**接收端拼接公式**:
```c
uint16_t received = ((uint16_t)crc_hi << 8) | crc_lo;
```

## 5. 错误处理

### 5.1 错误类型与处理对照表

| 错误类型 | 发生时机(FSM 状态) | 处理动作 | 数据损失 |
|---|---|---|---|
| 收到非 `0xAA` 字节 | WAIT_HEADER1 | 静默丢弃,留在当前状态 | 0 字节 |
| 收到非 `0x55` 非 `0xAA` 字节 | WAIT_HEADER2 | 回 WAIT_HEADER1 | 0 字节(尚未开始解析帧) |
| 收到 `0xAA` 字节 | WAIT_HEADER2 | **自环**,留在当前状态(可能是新帧头开始) | 0 字节 |
| LEN = 0 或 LEN > 64 | WAIT_LEN | 记 ERROR,resync 到 WAIT_HEADER1 | 1 帧 |
| payload 数据损坏 | READ_PAYLOAD | (不立即检测,等 CRC 阶段) | — |
| CRC 不匹配 | DELIVER | 记 ERROR,丢弃整帧,resync | 1 帧 |
| 收到 unexpected 字节 | DELIVER(瞬态) | 记 ERROR,resync | 1 帧(异常情况) |

### 5.2 resync 机制

**核心思想**:任何错误路径都汇聚到一个安全起点(WAIT_HEADER1)。

```
所有 ERROR → WAIT_HEADER1(resync)
            ↓
        继续找下一个 0xAA 帧头
            ↓
        正常解析下一帧
```

**最坏代价**:丢 1 帧(64 字节最大数据)。下一帧能正常解析,FSM 永不卡死。

### 5.3 LEN 合理性检查(防灾难传播)

WAIT_LEN 状态收到字节后,**立即检查范围**:

```c
if (len < 1 || len > 64) {
    log_error("LEN out of range");
    state = WAIT_HEADER1;  // resync
    return;
}
```

**为什么必须在 WAIT_LEN 立即检查**:
- 若不检查,损坏的 LEN(如 0xFF = 255)会让 FSM 在 READ_PAYLOAD 读 254 字节才进 CRC,中间会吞掉好几帧正常数据 → **灾难传播**
- 早期检查把灾难影响压缩到只丢 1 帧
- 体现 **Fail Fast** 原则

### 5.4 日志规范

错误日志应包含足够上下文,**让运维不读代码即可定位问题**:

```
[ERROR] LEN out of range: 80, expected [1, 64]
[ERROR] CRC mismatch: computed=0x5CA0, received=0xFF5B
```

**反例**(不推荐):

```
[ERROR] Bad frame                    ← 不知道哪里错了
[ERROR] CRC error                    ← 不知道差多少
```

## 6. 完整帧示例

### 6.1 心跳包(LEN=1)

```
完整帧:    AA 55 01 03 40 21
            │   │   │   │   │   │
            └帧头┘   │   │   └CRC┘
                    │   └TYPE=心跳
                    └LEN=1

CRC 计算:  CRC16({01, 03}) = 0x2140
字节序:    CRC_LO=0x40, CRC_HI=0x21
帧长:     6 字节(最小帧)
```

### 6.2 BH1750 光照帧(LEN=3)

```
完整帧:    AA 55 03 02 01 90 A0 5C
            │   │   │   │   │   │   │   │
            └帧头┘   │   │   └─┬─┘   └CRC┘
                    │   │     └ payload = 光照值 0x0190 = 400 lux
                    │   └ TYPE=BH1750 光照
                    └ LEN=3

CRC 计算:  CRC16({03, 02, 01, 90}) = 0x5CA0
字节序:    CRC_LO=0xA0, CRC_HI=0x5C
帧长:     8 字节
```
